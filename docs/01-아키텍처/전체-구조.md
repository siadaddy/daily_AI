# 전체 구조

세 개의 코드베이스가 한 저장소(`project_vibe_coding`)에 공존하지만, 배포/실행 방식이 완전히 다르다.

| 코드베이스 | 역할 | 실행 방식 |
|---|---|---|
| `src/` | Next.js 16 웹앱 (App Router) | Vercel 배포 |
| `pipeline/` | Python AI 파이프라인 — Naver 뉴스 수집, Gemini/Groq/HF로 카드뉴스·블로그·이미지 생성, Supabase 저장 | GitHub Actions `pipeline-daily.yml`, 매일 06:00 KST |
| `ai-crew/` | 독립 실행 뮤직 큐레이터 — `public/music/music_*.json` 생성 | 수동 실행 |

## 왜 이렇게 되어 있는가

세 코드베이스는 **의도적으로 분리되어 있다.** `pipeline/`과 `ai-crew/`는 `music_curator.py`,
`gemini_client.py`, `agent_memory.py` 같은 동일한 파일명을 쓰지만 로직이 다르다 — pipeline의
음악 큐레이터는 Supabase에 쓰고, ai-crew는 `public/music/`에 JSON 파일로 쓴다
([[pipeline-agents]], [[ai-crew-개요]] 참고). 한쪽을 고쳤다고 다른 쪽도 고쳐야 하는 건 아니다.

## 연결 지점

- `pipeline/` → Supabase(`news_cards`, `card_news`, `articles`, `news_trends`, `agents`, `logs`,
  `agent_memories`) 에 쓰기 → `src/`가 Server Component에서 읽어 렌더링.
- `ai-crew/` → `public/music/music_*.json` 파일 생성 → `src/components/music/MusicUniverse.tsx`가
  iframe으로 감싼 `/music/music.html`(Three.js)이 이 JSON을 읽음. Supabase를 거치지 않는
  유일한 데이터 경로.
- `src/`의 `/api/reports/generate`는 pipeline이 아니라 **웹앱 자체가** Vercel Cron으로
  OpenAI를 호출해 `weekly_reports`를 생성한다 — pipeline과는 독립적인 별도 흐름.
